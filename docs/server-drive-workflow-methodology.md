# Server-Drive Workflow 方法论

> 从 Kevlar MCP Server 提取的架构模式：**AI 只做理解与表达，不做流程与状态管理。**

---

## 核心原则

**Prompt 负责"为什么做"和"做成什么样"，Tool 负责"做什么"和"怎么做"，State Machine 负责"现在做到哪了"。**

---

## 四条可复用的模式

### 一、状态机托管在服务端，不在 system prompt 里

**问题：** 传统做法是在 SYSTEM_PROMPT 中写死流程规则（"先问 X，再问 Y…"），让 AI 自己推进。但 AI 会遗忘步骤、跳步、幻觉状态。

**做法：** 每个多步流程对应一个独立的 wizard 工具，服务端维护状态机（如 `ageRange → interests → traits → platform → finalConfirm`）。每次调用返回 `sessionId` + `state` + `nextStep`。宿主 AI 只需展示工具返回的信息给用户，再将用户回复传回同一工具。

**可复用的实现要点：**
- 定义 `WizardState` 枚举 + `SessionData` 接口
- 每个步骤有明确的输入校验和状态转换逻辑
- AI 只负责"把用户自然语言映射到当前步骤的字段值"

### 二、字段校验和值推导在服务端完成

**问题：** 让 AI 既做自然语言提炼又做字段校验，结果不可靠。

**做法：** 分层处理——
- **AI 层**：负责"优雅的映射"，如用户说"一个爱刷小红书的 25 岁设计师" → tags: `["小红书", "设计"]`
- **服务端校验层**：负责严格的格式校验（年龄范围是否合法、必填字段是否完整）
- **服务端推导层**：负责规则的确定性推导（根据已有字段推断派生字段，如文化背景、立场、盲区）

### 三、高风险操作通过"确认向导"而非 AI 判断

**问题：** 让 AI 语义理解"用户是否真的想删除"，容易误判。

**做法：**
1. 工具先展示影响范围（预览变更）
2. 要求用户回复固定的确认语（如 `确认删除{人设名}`）
3. 只有确认语精确匹配才执行

**要点：** 对不可逆操作，要求精确匹配确认字符串，而非 AI 语义理解。

### 四、工具即契约（Tool as Contract）

**问题：** AI 需要记忆流程细节、字段格式、下一步逻辑，prompt 膨胀且不一致。

**做法：** 每个 wizard 工具的参数列表就是该流程的契约接口——
- AI 不需要记住"下一问应该问什么"——工具的返回值告诉它
- AI 不需要猜测"字段格式是什么"——参数 schema 定义了
- 流程的"协议层"由工具 schema 定义，而非 prompt 描述

---

## 架构对比

| 维度 | 传统 Prompt-Driven | Server-Drive Workflow |
|------|-------------------|----------------------|
| 流程状态 | system prompt 中描述 | 服务端状态机维护 |
| 字段校验 | AI 自行判断 | 服务端参数 schema 校验 |
| 值推导 | AI 推断（不一致） | 服务端确定性推导 |
| 高风险确认 | AI 语义理解 | 精确确认语匹配 |
| 协议契约 | prompt 描述（易腐） | 工具 schema（可验证） |

---

## 适用场景

- MCP / Function Calling 架构下的多步交互流程
- 需要人机协作确认的操作（创建、删除、修改配置）
- 涉及状态持久化的向导式流程
- 高风险操作的二次确认

## 不适用场景

- 纯文本生成、翻译、总结等无状态任务
- 不需要分步交互的单次查询
