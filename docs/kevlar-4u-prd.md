# Kevlar-4u 内容风控与智能联想引擎需求文档 (PRD)

## 1. 文档概述

### 1.1 背景

Kevlar-4u 作为一款开源的内容风控与评论区模拟器，旨在帮助创作者在发布内容前进行安全排雷。互联网黑话、谐音变体层出不穷，且长文本下多角色审查易出现"人格串味"。

为了在第一阶段（本地开源 MCP）保持轻量、隐私，并为第二阶段（商业化动态订阅）打下架构基础，特制定本核心技术需求文档。

### 1.2 核心目标

- **零体积膨胀**：将词库与规则体积控制在 5KB 以内，摆脱百万级静态死词库。
- **主动演化联想**：通过动态规则与红队思维链，使 AI 具备对未知新词、谐音、拆字的泛化联想能力。
- **完美人格隔离**：在单次 Prompt 交互中，并行模拟多个独立立场的审查员，杜绝偏见协同。

## 2. 核心系统架构与数据流设计

本系统由本地轻量规则引擎 (DAO) 与 大模型协同交互层 (LLM Agent) 双层驱动。

### 2.1 整体数据流向

- **输入阶段**：用户提交待发布文本 → 触发现阶层 L1/L2 本地过滤。
- **处理阶段**：系统提取规则补丁 → 组装包含「演化规则」、「模拟并行代码块」与「红队思维链」的黄金 Prompt。
- **输出阶段**：LLM 按照伪代码格式隔离输出各角色审查结论 → 汇总生成沙盒测试报告。

## 3. 功能需求详述

### 需求一：基于规则化描述的轻量级规则库 (rules.json)

**需求目的**：放弃传统大体积敏感词典，改用模式描述与最小化词根库，支持本地高并发检索并预留云端热下发接口。

**业务逻辑**：

1. 本地存储一份结构化的 `rules.json`（总体积 <5KB）。
2. 包含 `association_patterns`（高危模式匹配）、`evolution_strategies`（演化路径定义）以及 `risk_roots`（高频初始词根）。

**可执行 JSON 样例结构**：

```json
{
  "core_rules": {
    "association_patterns": [
      { "pattern": "颜色+身体部位", "risk_type": "涉黄风险" },
      { "pattern": "食材+异常修饰", "risk_type": "黑话暗语" },
      { "pattern": "人名+负面标签", "risk_type": "人身攻击" }
    ],
    "evolution_strategies": ["缩写演化", "谐音演化", "拆字演化", "Emoji嵌入"],
    "risk_roots": [
      { "word": "木耳", "variants_check": ["粉", "黑", "白", "红"] },
      { "word": "菊花", "variants_check": ["XX", "爆"] }
    ]
  }
}
```

**工程要求 (DAO 层)**：

- 系统启动时，必须将 `variants_check` 平铺展开存入内存 HashSet（一级索引：实现 O(1) 极限拦截）。
- 将 `risk_roots` 存入以 `word` 为 Key 的 HashMap（二级索引：用于运行时拉取误读方向与处置建议）。

### 需求二：大模型"主动联想"与红队思维链植入

**需求目的**：通过 Prompt 注入，强行激活 LLM 的"恶意解读"与"路径推演"本能。

**规则设计**：

在拼接发送给 LLM 的核心 Prompt 时，必须强制嵌入以下【联想四步法】与【反向测试指令】。

**Prompt 模板规范**：

```markdown
# 角色设定
你是网络文化风控专家与恶意舆论红队测试员。在审查用户输入时，必须严格执行以下逻辑：

## 一、【联想四步法】（强制逐词执行）
1. 词根提取：提取输入文本中的核心语素或潜在核心概念（如食物、身体、谐音字）。
2. 模式匹配：检查是否命中 [颜色+身体部位] 或 [食材+异常修饰] 等高危模式。
3. 演化联想：主动推演其缩写形式、谐音形式（如同音替换）、拆字形式（如艹、逼）是否在当下亚文化社区具备恶意。
4. 语境检查：判断当前语境是属于正常表达（如正常食材描述）还是触发了暗语恶意。

## 二、【反向测试红队指令】
假设你是一个极端恶意的匿名网友，为了故意"冲"这段内容，你会如何操作：
- 哪些词可以被单独拎出来断章取义？
- 哪些谐音/缩写可以被用来做恶意群体的对线表情包？
- 如果字面意思和实际意思产生语义断裂，最坏的引战方向是什么？

执行要求：宁可误报，不可漏报。即使看起来正常的词，也要给出你的演化路径。
```

### 需求三：宿主辅助伪并行架构（多角色隔离）

**需求目的**：在单次 Prompt 交互中模拟 9 种不同基本盘立场（如独立女性、理性男性、务实职场人等），防止角色之间产生记忆交叉、观点顺从或"人格串味"。

**技术方案**：

利用 LLM 对代码逻辑极其刻板的执行特性，使用「伪代码框架」+「隔离墙指令」约束大模型的输出时序。

**Prompt 输出格式控制规范**：

```markdown
# 并行模拟执行规范
你将同时扮演 9 个相互隔离、毫无交集的独立审查员（如独立女性视角、务实职场人等）。
请严格按照以下 Python 伪代码的逻辑和结构顺序执行，不得跳过任何一个 Agent 的思考：

```python
# 伪代码：模拟独立Agent并行执行并 append 结果
audit_results = []

# Agent 1: [独立女性视角审查员] 执行独立审查
agent1_output = 独立女性视角审查(content)
audit_results.append(("独立女性视角", agent1_output))

# Agent 2: [理性男性视角审查员] 执行独立审查
agent2_output = 理性男性视角审查(content)
audit_results.append(("理性男性视角", agent2_output))

# ... 依次执行其余角色

# 汇总层（仅在所有人独立完成后）
final_report = merge_all(audit_results)
```

**执行强制模板**：

```
═══════════════════════════════════════
▶ 审查员 1 号：[角色名称]
═══════════════════════════════════════
【独立声明】我声明我的评估不参考任何其他角色。
[独立思考过程（必须包含联想四步法）]
[结论：🟢安全 / 🟡低危提示 / 🔴高危建议]

（以此类推，严格输出各角色模块...）

═══════════════════════════════════════
▶ 最终风控合并报告
═══════════════════════════════════════
[汇总冲突点与极限排雷建议]
```

## 4. 非功能性需求与未来商业化预留

### 4.1 性能与隐私需求 (本地版)

- **网络隔离**：本地开源版（MCP 服务）运行期间，所有词根提取与 LLM 组织工作必须在用户本地或其配置的私有 LLM 节点完成，严禁上传文本至未经授权的第三方服务器。
- **处理延迟**：本地 HashMap/HashSet 规则检索耗时需控制在 ≤5ms 级别。

### 4.2 商业化扩展性设计 (面向第二阶段付费订阅)

为了确保一阶段（本地开源）到二阶段（付费订阅）的无缝切换，接口设计必须实现**统一数据访问层隔离**：

1. 须定义统一规则仓库接口 `IRuleRepository`。
2. **第一阶段实现**：实例化 `LocalJsonRuleRepository`，直接读取本地预置的脱敏 `rules.json`。
3. **第二阶段实现**：当用户在前端输入付费 Subscription Key 后，系统混入 `RemoteSubscriptionRuleRepository`。系统每 6 小时发起 HTTPS 静默检查，云端下发经 **AES-GCM** 加密的最新热点 `risk_roots`（补丁包），在**运行时内存中热解密加载**，绝对不允许明文落盘。

## 5. 需求验收标准 (Definition of Done)

1. **体积测试**：本地 `rules.json` 规则文件总大小不得超过 5KB。
2. **联想测试**：输入正常词汇的擦边变体（例如"粉耳"、"局花"），系统能够通过【联想四步法】正确逆向推演出"木耳"、"菊花"词根，并标明潜在涉黄风险路径。
3. **隔离测试**：单次模拟输出中，各审查员角色的思考逻辑不得使用"正如前一位审查员所说"或受其结论左右的词句，必须严格遵循伪代码规定的物理隔离分块输出。
