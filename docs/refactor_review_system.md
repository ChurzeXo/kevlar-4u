# 重构需求文档
# reviewDispatcherPrompt v2 & reviewPersonaPrompt v2

> 本文档基于逻辑审计报告（11条审计结论）输出，按优先级排序。
> 每条需求对应原审计编号，可追溯。

---

## 一、重构原则

1. **流程完整性优先**：所有异常路径必须有明确的降级行为，不允许静默失败
2. **Don't Make Me Think**：所有对用户可见的输出，必须使用行动语言，不使用系统语言
3. **职责单一**：调度层只管「谁来评论」，子代理只管「怎么评论」，边界不得模糊
4. **最小改动**：在满足以上三条原则的前提下，保持原有流程结构不变

## 二、主AI调度层重构需求（reviewDispatcherPrompt v2）

---

### D-1 角色库状态前置检查
**对应审计**：A-01
**优先级**：P0
**原问题**：流程在匹配前没有检查角色库状态，角色库为空时流程卡死

**重构要求**：

在「一、评审员匹配」最前面新增角色库状态检查步骤，优先于任何匹配逻辑执行：

```
角色库检查（优先执行）：
- 若角色数量 = 0：
  停止匹配流程，提示用户：
  「当前还没有可用的评审员，请先创建至少一个角色，再提交内容进行评论。」
  
- 若角色数量 = 1-2：
  跳过推荐步骤，直接展示全部可用角色，说明：
  「当前只有 N 位评审员可用，已为你展示全部角色，请确认是否使用。」
  
- 若角色数量 ≥ 3：
  执行正常匹配推荐流程
```

---

### D-2 推荐理由结构化
**对应审计**：A-02
**优先级**：P2
**原问题**：「基于内容特征」过于抽象，推荐理由质量无法保证

**重构要求**：

将「推荐理由须基于内容特征给出，不得随机指派」替换为：

```
推荐理由必须包含以下三项中的至少两项：
① 内容类型匹配：该角色的兴趣方向与内容主题的关联
② 平台匹配：该角色的常用平台与内容分发渠道一致
③ 风格匹配：该角色的性格特质与内容风格的关联

示例格式：
「角色A（推荐理由：内容是数码类测评，该角色兴趣方向包含科技数码，且常用平台为小红书，与本次内容分发渠道一致）」
```

---

### D-3 自选列表展示格式
**对应审计**：A-03
**优先级**：P3
**原问题**：自选时只展示角色名，用户无法快速判断

**重构要求**：

将「列出剩余评审员供用户勾选」替换为：

```
自选列表展示格式（每个角色一行）：
[角色名] · [常用平台] · [一句话描述]

示例：
- 挑剔的数码买家 · 小红书 · 习惯先看评论区，对没有真实使用感受的内容直接划走
- 职场效率控 · LinkedIn · 只关注能解决实际问题的内容，对空泛的方法论没有耐心
```

---

### D-4 子代理超时与失败处理
**对应审计**：A-04
**优先级**：P0
**原问题**：子代理失败时调度层无处理机制，整个报告卡死

**重构要求**：

在「二、子代理协调」中新增失败处理规则：

```
子代理执行异常处理：
- 若某个子代理返回错误或超时未响应：
  标记该评审员状态为「执行失败」
  不阻塞其他子代理的执行
  
- 汇总时在对应评审员位置输出：
  「[角色名] · 本次执行未能完成，请重新提交内容后再试。」
  
- 若所有子代理均失败：
  告知用户「本次评论全部未能完成」，并提示重试，不生成空报告
```

---

### D-5 报告文件命名规范
**对应审计**：A-05
**优先级**：P3
**原问题**：文件名未定义，多次保存后桌面文件无法区分

**重构要求**：

在「三、报告汇总与交付」保存逻辑中新增命名规则：

```
文件命名格式：
kevlar_{内容前10个字（去除标点）}_{YYYYMMDD_HHMM}.html
kevlar_{内容前10个字（去除标点）}_{YYYYMMDD_HHMM}.md

示例：
kevlar_我做了一套适合内容创作_20250519_1430.html

若内容前10个字包含特殊字符或空格，统一替换为下划线。
```

## 三、子代理执行层重构需求（reviewPersonaPrompt v2）

---

### P-1 「未打开」场景专用输出模板
**对应审计**：B-01
**优先级**：P0
**原问题**：「未打开」结论产生后，后续输出格式要求填写需要深读全文才能完成的字段，导致逻辑撕裂和幻觉输出

**重构要求**：

在「四、诊断输出格式」中新增「未打开」专用短链路模板，当钩子判断结论为「未打开」时，跳过标准输出模板，使用以下格式：

```
─────────────────────────────
【角色名】

评分
钩子强度     ■□□□□□□□□□  X/10  「大部分人会直接划走」

读者视角
「没有打开，无法判断内容在说什么」

钩子判断
未打开 · 原因：XXX（具体说明是标题、第一句还是缺乏触发动机）

核心建议
🔧 XXX（必须给出一条具体可执行的改进建议，针对钩子问题）

─────────────────────────────
```

说明：「未打开」时不输出槽点、亮点、行动意向、行动阻力，因为角色没有完整阅读内容，这些字段无法真实填写。

---

### P-2 读者视角前置
**对应审计**：B-02
**优先级**：P1
**原问题**：诊断价值最高的「读者视角」被埋在评分之后

**重构要求**：

调整「四、诊断输出格式」的字段顺序，将「读者视角」移至第一位：

```
─────────────────────────────
【角色名】

读者视角                        ← 前置，第一眼看到
「我以为这是在说：XXX」

评分                            ← 移至读者视角之后
钩子强度     ■■■□□□□□□□  3/10
信息清晰度   ■■■■■□□□□□  5/10
行动转化     ■■■■■■■□□□  7/10

钩子判断
...
```

---

### P-3 评分附加行动预测语言
**对应审计**：B-03
**优先级**：P1
**原问题**：纯数字评分是系统语言，用户需要自行解码，违反 Don't Make Me Think 原则

**重构要求**：

每个评分维度后强制附加一句口语化预测语言，规则如下：

```
钩子强度预测语言：
- 1-3分：「大部分人会直接划走」
- 4-6分：「部分人会停下来，但动力不足」
- 7-10分：「容易吸引目标读者打开」

信息清晰度预测语言：
- 1-3分：「读者大概率误解了你的意图」
- 4-6分：「读者理解了部分，但核心信息不够清晰」
- 7-10分：「读者基本理解了你想表达的内容」

行动转化预测语言：
- 1-3分：「有人看完，但不会有任何行动」
- 4-6分：「少部分人会有行动，但阻力明显」
- 7-10分：「目标读者有较高概率完成行动」

输出示例：
钩子强度     ■■■□□□□□□□  3/10  「大部分人会直接划走」
信息清晰度   ■■■■■□□□□□  5/10  「读者理解了部分，但核心信息不够清晰」
行动转化     ■■■■■■■□□□  7/10  「目标读者有较高概率完成行动」
```

---

### P-4 槽点与亮点数量上限
**对应审计**：B-04
**优先级**：P2
**原问题**：槽点无数量上限，核心问题被细枝末节稀释

**重构要求**：

在「五、注意事项」中新增数量约束：

```
槽点输出规则：
- 最多输出 3 条
- 按严重程度倒序排列（最影响转化的排第一）
- 每条必须包含三层颗粒度：哪里 + 什么问题 + 为什么是问题

亮点输出规则：
- 最多输出 2 条
- 若无亮点，说明「未发现明显亮点，但 X 部分没有拖后腿」
- 不得省略此项
```

---

### P-5 最终判定收束为可测量的转化动作
**对应审计**：B-05
**优先级**：P2
**原问题**：「在情感上想做什么」过于开放，AI 容易输出无法衡量的空话

**重构要求**：

将最终判定的输出规则替换为：

```
行动意向规则：
必须从以下动作中选择一个最真实的，不允许使用其他表达：
点赞 / 转发 / 收藏 / 评论 / 划走 / 关注作者

行动阻力规则：
必须从以下类型中选择一种，并说明具体原因：
① 没有对话锚点：「文章是个句号，我接不上话」
② 评论成本过高：「我想说什么，但脑子里组织不出一句话」
③ 传播场景缺失：「想转发，但想不到发给谁」
④ 信任阈值未过：「内容不错，但还不够信任作者」
⑤ 收益感知模糊：「点赞对我有什么意义」

输出示例：
· 行动意向：收藏
· 行动阻力：传播场景缺失——读完感觉有用，但想不到现在能用在哪里，收藏了也不会回来看
```

---

### P-6 角色描述加载失败阻断机制
**对应审计**：B-06
**优先级**：P0
**原问题**：关键字段缺失时子代理静默失败，用默认性格继续执行破坏人设前提

**重构要求**：

在「一、执行前准备」中新增字段校验与阻断规则：

```
关键字段校验（加载后立即执行）：

必须校验以下字段是否存在且非空：
- 性格特质（至少1条）
- 常用平台
- 盲区

若以上任一字段缺失：
立即终止执行，向调度层返回错误信息：
「[角色名] 角色描述不完整，缺少以下字段：XXX，无法执行评论任务。」

不得使用默认值或模型自身预设继续执行。
```

## 四、重构优先级汇总

| 优先级 | 需求编号 | 所属层 | 核心变更 |
|--------|----------|--------|----------|
| P0 | D-1 | 调度层 | 角色库状态前置检查，空库时终止并提示 |
| P0 | D-4 | 调度层 | 子代理失败标记跳过，不阻塞整体报告 |
| P0 | P-1 | 子代理 | 「未打开」专用短链路输出模板 |
| P0 | P-6 | 子代理 | 关键字段缺失时阻断执行并返回错误 |
| P1 | P-2 | 子代理 | 读者视角前置至输出第一位 |
| P1 | P-3 | 子代理 | 每项评分附加口语化行动预测语言 |
| P2 | D-2 | 调度层 | 推荐理由结构化，至少说明两项匹配依据 |
| P2 | P-4 | 子代理 | 槽点≤3条，亮点≤2条，按严重程度排序 |
| P2 | P-5 | 子代理 | 行动意向收束为6个具体动作，行动阻力对应5种类型 |
| P3 | D-3 | 调度层 | 自选列表附加一句话描述和适用平台 |
| P3 | D-5 | 调度层 | 报告文件名自动生成，含内容摘要和时间戳 |

## 五、重构后流程对比

### 调度层流程变化

```
原流程：
用户提交内容 → 匹配评审员 → 用户确认 → 子代理执行 → 汇总展示 → 询问保存

新流程：
用户提交内容
  → [新增] 角色库状态检查（空库终止，不足3个直接展示全部）
  → 匹配评审员（推荐理由结构化）
  → 用户确认（自选列表附加描述）
  → 子代理执行（[新增] 失败标记跳过机制）
  → 汇总展示
  → 询问保存（[新增] 自动命名规范）
```

### 子代理流程变化

```
原流程：
加载角色 → 内部推理（6步）→ 评分 → 诊断输出

新流程：
加载角色
  → [新增] 关键字段校验（缺失则阻断返回错误）
  → 内部推理（6步）
  → 步骤2钩子判断
      ├── 未打开 → [新增] 短链路输出模板（跳过后续推理）
      └── 已打开 → 继续步骤3-6
  → 评分（[新增] 附加行动预测语言）
  → 诊断输出
      ├── [调整] 读者视角前置至第一位
      ├── [新增] 槽点≤3条，亮点≤2条
      └── [收束] 行动意向限定6个动作，行动阻力限定5种类型
```
