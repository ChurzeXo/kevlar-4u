# Kevlar 提示词注入安全审计报告（本地威胁模型）

**审计日期**：2026-05-22
**部署场景**：纯本地单用户（如 Claude Desktop、Cursor 等本地 MCP 客户端）
**威胁假设**：无外部攻击者；用户即系统唯一操作者

---

## 威胁模型声明

Kevlar 是**纯本地工具**，不存在多租户、用户输入共享、外部数据源自动摄取等场景。因此传统提示词注入攻击的两个核心前提不成立：

1. **不存在攻击者**：唯一操作者是用户自己，无人需要"注入"恶意指令给另一个用户
2. **不存在提权路径**：用户已拥有其本地机器、环境变量、文件系统的全部权限

本审计在此前提下重新定级。

---

## 防御层回顾摘要

| 防御层 | 定级 | 说明 |
|--------|------|------|
| L1 结构化隔离 | 🟢 **够用** | 无外部攻击者，边界环绕是健壮性而非安全性 |
| L2 动态护栏 | 🟢 **不需要** | 无第三方输入共享，前置/后置过滤无实际收益 |
| L3 运行时沙箱 | 🟡 **部分需要** | 凭证分离已做好；进程隔离对本地单用户场景收益低 |
| L4 主动确认审计 | 🟢 **够用** | 销毁操作 HITL 已到位；评测 HITL 是 UX 问题 |

---

## 评估结论

现有防御**对本地使用场景已经足够安全**。下面是逐项分析与真实风险定级。

---

## L1 结构化隔离

### L1-1 [健壮性] 用户内容裸嵌入，无语义转义

**之前定级**：CRITICAL  
**本地模型定级**：🟢 **P4（可忽略）**

**分析**：
- Orchestration 模式下，用户内容嵌入 `<content>` 标签。如果内容包含 `</content>`，prompt 结构会断裂。
- **但用户是在给自己的 AI 发送 prompt**，断裂的结果只是 AI 输出混乱，用户自己纠正即可。
- 不存在攻击者利用这个断裂窃取数据或提权。

**建议**：作为代码健壮性问题标记为 Known Issue，无需紧急修复。如果未来开放编排模式给其他不信任的客户端，需要修。

### L1-2 [健壮性] Orchestration 模式无 persona 间隔离

**之前定级**：MEDIUM  
**本地模型定级**：🟢 **P4（可忽略）**

**分析**：单 prompt 中 persona 之间可能串味。这是编排模式自身的精度限制，用户选择使用编排模式就已知精度下降。不存在安全后果。

### L1-3 [不需要] Persona 系统提示词注入传播

**之前定级**：MEDIUM  
**本地模型定级**：🟢 **关闭**

**分析**：Persona 的内容本身就是用户自己定义的。用户给自己的 AI 创建角色并用于评测——这是正常功能，不是攻击。

---

## L2 动态护栏

### L2-1 [不需要] 零前置输入护栏

**之前定级**：CRITICAL  
**本地模型定级**：🟢 **关闭**

**分析**：前置护栏（注入检测、意图分类）解决的是"不可信用户攻击系统"的场景。本地场景的唯一用户自己对自己负责。没有任何安全收益。

### L2-2 [不需要] 零后置输出护栏

**之前定级**：CRITICAL  
**本地模型定级**：🟢 **关闭**

**分析**：后置护栏（DLP、凭证泄露检测）解决的是"AI 被攻破后向不可信方吐出数据"的场景。本地场景中输出直接返回给用户自己。用户看到自己的 API key 在输出中的概率极低，且看到了也没有安全后果。

**注释**：API key 已通过 `maskApiKey()` 在日志中遮盖，且 key **不会**作为 prompt 的一部分传给 LLM（仅作为 HTTP header 发送给 API provider）。不存在 key 通过 LLM 输出泄露的路径。

---

## L3 运行时沙箱与凭证解耦

### L3-1 [已验证] 凭证分离设计合理

✅ 维持原判。API keys 仅从环境变量读取，不落盘，不在工具参数中传递，不在日志中明码显示。

### L3-2 [不需要] 无进程/容器沙箱

**之前定级**：MEDIUM  
**本地模型定级**：🟢 **关闭**

**分析**：沙箱防止的是"恶意代码提权逃逸"和"多租户串扰"。单用户本地无此需求。如果用户不信任 Kevlar 这个 Node.js 进程，不应该在本地运行它。

### L3-3 [低] Direct API 发送 `anthropic-dangerous-direct-browser-access` 头

**之前定级**：LOW  
**本地模型定级**：🟢 **P4（低优先级）**

**分析**：该头在非浏览器环境中无实际效果。可以去掉以减少 API 请求的异常特征，但无安全必要。

### L3-4 [了解即可] Direct API 模式发送内容给第三方

这是 Kevlar 的核心功能（调用 Anthropic/OpenAI/Ollama 进行评测），用户选择该模式时应知悉内容和 persona 提示词会发送给对应的 API provider。

建议在 README 或文档中注明此行为，让用户知情即可。

---

## L4 主动确认与闭环审计

### L4-1 [已验证] 销毁操作 HITL 机制完善

✅ 维持原判。

### L4-2 [UX 问题] 评测执行无 HITL

**之前定级**：MEDIUM  
**本地模型定级**：🟡 **P3（UX 改进）**

**分析**：用户确认后立即执行评测，没有成本预览或二次确认。这是 UX 问题而非安全问题——如果用户在 Direct API 模式下不小心评测了 100KB 内容，会产生 token 费用。建议在 Direct API 模式下显示预估 token 消耗和成本提示。

### L4-3 [非问题] `isAffirmative()` 子串匹配

**之前定级**：LOW  
**本地模型定级**：🟢 **关闭**

**分析**：误确认的后果是 wizard 进入下一步，并非执行破坏性操作。最终的删除/修改操作仍有独立的精确确认短语把关。

### L4-4 [低] 审计日志仅在 stderr

**之前定级**：LOW  
**本地模型定级**：🟡 **P3（可选）**

**分析**：日志持久化对单用户场景价值有限。如果用户需要调试或追踪历史，可以后续添加文件输出选项。

---

## 最终判断

| 维度 | 状态 |
|------|------|
| API key 安全性 | ✅ 设计合理，无暴露路径 |
| 文件系统安全 | ✅ 有路径遍历防护和校验 |
| 内容泄露风险 | 🟡 用户应知悉 Direct API 模式下内容发送给第三方 |
| 注入攻击面 | 🟢 本地场景无有效攻击者 |
| 数据完整性 | ✅ HITL 覆盖所有破坏性操作 |
| 资源控制 | ✅ 并发锁、token 预算检查到位 |

**结论：对纯本地单用户场景，Kevlar 已提供足够的安全防护。现有审计中的大多数问题在 SaaS/多租户场景下成立，但本地场景不适用。**

如果未来部署模式变为：
- 多用户共享 Kevlar 服务
- 接入不可信的第三方数据（网页抓取、邮件读取等）
- 作为公共服务对外暴露

则需要按第一版报告中的 P0/P1 标准进行加固。
